# Findings

## P1 — PR #13 仍以废弃的 full-disk authorization 作为跨仓库授权模型

* **Severity:** P1
* **Reference:** PR #13 `b27885da8767126ec6ad376503b26a277492d032`
  `src/cli/mcp/tools.ts::{resolveFullDiskRepoAlias,targetRepoRoot}`
  `tests/cli/mcp-tools.test.ts`
* **Observed evidence:** `resolveFullDiskRepoAlias()` 仅在 `isFullDiskRead(ctx)` 为真时运行。它既可以接受当前仓库外实际存在的相对路径，也可以进行全盘 discovery；一旦得到 alias，`targetRepoRoot()` 立即返回成功，跳过后续 repo 授权。相比之下，current main 会 canonicalize 目标，并要求它是 registered repo，或位于显式 discovery root 下的 adopted repo，且再次验证 adopted 状态。current main 的 setup 还会把 legacy `fullDiskRead:true` 迁移为 `workspaceReader`，最终持久化 `fullDiskRead:false`。([GitHub][1])
* **Observed evidence:** PR #13 的测试仍明确构造 `fullDiskRead:true`，并把 `.env` 从 `POLICY_DENIED` 改为“允许读取后再 redact”。它也把跨仓库 discovery/read 定义为“under full-disk authorization”。这不是 current-main deny-boundary 的等价行为：redaction 不等于拒绝读取。([GitHub][2])
* **Why it matters:** 该路径绕过了 PR #16 后的 registered/adopted repo 边界，重新把“能扫描磁盘”当成“有权读取该 repo”。这会扩大 MCP reader 的有效访问范围，并与 `workspaceReader`、registered repos 和 `fullDiskRead:false` 的主线方向直接冲突。
* **Concrete fix / verification:** 删除 `resolveFullDiskRepoAlias` 的授权语义。alias 只能解析到 registry 中 canonical、仍 adopted 的记录；解析成功后仍必须进入 current-main `targetRepoRoot` 的完整授权路径。`.env`、`.git`、`.ssh`、`secrets`、`credentials`、`private`、`node_modules`、`dist`、`build`、`coverage` 等必须在 alias、绝对路径和 workspace-reader 三种入口上均返回 policy denial。current main 的 deny 集合已明确包含这些路径。([GitHub][3])

## P1 — PR #13 的 alias 解析不唯一、不具可审计性，也不是 fail-closed

* **Severity:** P1
* **Reference:** PR #13 `b27885d`
  `src/cli/mcp/tools.ts::{normalizeRepoSearchTerm,repoMatchesQuery,discoverySortWeight,resolveFullDiskRepoAlias}`
* **Observed evidence:** 输入被截断为 lowercase basename；匹配规则接受 basename 相等、basename substring 或完整路径 substring。结果再按 exact/prefix/substring 和路径字符串排序，随后调用 discovery `{limit:1}`，直接采用第一项。两个同名 repo、仅大小写不同的 repo、`app`/`app-server` 前缀碰撞，或路径中偶然含有 query 的 repo，都不会产生 ambiguity error。([GitHub][1])
* **Why it matters:** 在多个 workspace 或 case-insensitive 文件系统上，同一个 `repo_path` alias 可能静默路由到错误 repo。排序顺序不是授权身份，也不能形成可靠审计记录；目录增加、挂载位置变化或大小写变化都可能改变选中对象。
* **Concrete fix / verification:** 使用稳定 registry ID、canonical path 或显式唯一 alias。零匹配返回 `REPO_NOT_FOUND`，多匹配返回 `AMBIGUOUS_REPO`，并只报告无敏感信息的候选标识；绝不能自动挑第一项。增加 duplicate basename、prefix、substring、大小写碰撞、stale registry、unadopted repo、symlink/junction 和跨 discovery-root 测试。

## P1 — PR #15 当前 head 没有保留 PR #16 的完整 hook repo binding 与 isolation 实现

* **Severity:** P1
* **Reference:** PR #15 `3aa1dbb`
  `src/cli/hook/runtime.ts::runHook`
  `.ai/hooks/prompt-guard.sh`
  `assets/hooks/prompt-guard.sh`
  current-main fix `2acd7a4963a0975f15e9af099c4aa44f67180990`
* **Observed evidence:** current main 的 `runHook()` 通过 `resolveExplicitRepoRoot()` canonicalize `HOOK_REPO_ROOT` 与 cwd repo；两者冲突时以 `reason:'repo-root-mismatch'`、exit 0 静默跳过。PR #15 head 中的 `runHook()` 仍直接以 cwd 调用 `resolveRepoRoot()`，没有该 conflict-skip 分支。([GitHub][5])
* **Observed evidence:** PR #16 在 prompt-guard 中新增 bounded symlink-chain resolution、20-hop 上限、nearest-existing-path 和 `prompt_foreign_repo_root`，并在 think/plan 与 embedded-plan capture 入口遇到 sibling/foreign repo 时跳过当前 repo 的写入。PR #15 以 `e60a1d6` 为父节点，同时修改同一批 prompt-guard/runtime 文件，因此其当前 head 本身不包含这些后来加入的防护，也没有 current-main conflict-resolution 结果。([GitHub][6])
* **Why it matters:** 错误合并可能让从 sibling repo、显式冲突的 `HOOK_REPO_ROOT` 或 symlinked path 发出的 hook 在错误仓库中创建/修改 workflow 文件。这正是 PR #16 已关闭的 P1。
* **Concrete fix / verification:** 在 `98240c8` 上重新集成，并保留：

  * explicit `HOOK_REPO_ROOT`/cwd canonical conflict → exit 0、无副作用；
  * adopted sibling repo isolation；
  * 相对/绝对、多跳、dangling、循环 symlink handling；
  * think/plan 与 embedded-plan 两个 foreign-repo gate；
  * `.ai/hooks` 与 `assets/hooks` 的相同实现。
    必须新增 regression tests，而不是只依赖手工冲突解决。

## P1 — PR #15 对旧仓库缺省启用四个 hook 阶段，且 Stop 副作用发生在 reentrancy 检查之前

* **Severity:** P1
* **Reference:** PR #15 `3aa1dbb`
  `src/cli/hook/minimal-change-policy.ts::{DEFAULT_MINIMAL_CHANGE_POLICY,loadMinimalChangePolicy}`
  `.ai/hooks/stop-orchestrator.sh::{minimal_change_append_handoff,main flow}`
  mirrored `assets/hooks/stop-orchestrator.sh`
* **Observed evidence:** 缺少 policy 文件、缺少 `minimal_change` 字段或 JSON 解析失败时，loader 都回退到同一默认配置；该默认配置为 `mode:'advice'`，并启用 `session_context`、`prompt_advice`、`post_edit_observer`、`stop_review`。因此现有 adopted repo 无需显式迁移或 opt-in 就会改变四个生命周期阶段的行为。([GitHub][7])
* **Observed evidence:** Stop 路径先 refresh handoff、运行 minimal-change review，并用 `>>` 追加 `## Minimal Change Review`，之后才读取 `stop_hook_active`。`event_dedupe` 只阻止相同 report fingerprint 被重写，并不使 handoff section 幂等。([GitHub][8])
* **Why it matters:** recursive/reentrant Stop 仍会执行分析和持久化副作用；正常升级也会无提示启用新 advice。即使不 block agent，它仍可能修改 handoff、增加上下文输出或在重复 Stop 流程中产生重复/陈旧 section。
* **Concrete fix / verification:** 对缺失或 malformed 配置默认 `mode:'off'`；通过 migration/scaffold 显式 opt-in。把 `stop_hook_active` 检查移到 refresh/review/append 之前。handoff 使用有边界 marker 的 replace/upsert，而不是裸 `>>`。增加连续两次 Stop、`stop_hook_active:true`、相同 fingerprint、改变 fingerprint、无 Bun/jq、malformed report 的幂等与 fail-open 测试。

## P2 — PR #15 的默认 PostToolUse observer 对正常编辑路径偏重且可能产生噪声

* **Severity:** P2
* **Reference:** PR #15 `3aa1dbb`
  `src/cli/hook/route-registry.ts` PostToolUse `edit` route
  `src/cli/hook/minimal-change-signals.ts::{git,collect/edit signal path}`
  `.ai/hooks/minimal-change-observer.sh`
* **Observed evidence:** 每次 `Edit|Write` 都会运行 `minimal-change-observer.sh`。单次分析至少可能执行 `rev-parse`、`diff --name-status`、`diff --numstat`、tracked-file 查询，并在 manifest 情况下执行更多读取；每个 Git 子进程为同步调用，单独设置 1500ms timeout。observer 失败时虽然 exit 0，但会向 stderr 输出 `minimal-change observer skipped: ...`。([GitHub][9])
* **Why it matters:** 在大型仓库、网络文件系统或频繁小编辑工作流中，多个同步 Git 调用会累积延迟；fail-open 仍然可能把环境性错误反复打印给 agent/user。
* **Concrete fix / verification:** 默认关闭 post-edit observer，或按 session/debounce 批处理；复用 repo root/base-ref，限制 manifest 分析频率。预期缺失、timeout 和非 Git 场景应静默。为单次编辑和 50 次连续编辑建立延迟预算与 stderr-zero smoke test。

## P2 — PR #15 不适合作为 `0.7.5` 的隐式 patch 内容，且 packaged runtime 尚未在 current main 上证明

* **Severity:** P2
* **Reference:** PR #15 `3aa1dbb`；`package.json`; current main release line
* **Observed evidence:** PR #15 是 41 文件、约 4,084 行新增的行为型功能，覆盖 SessionStart、UserPromptSubmit、PostToolUse 和 Stop，并改变缺省 policy。其父基线仍是 `0.7.4`，而 current main 已标记为 `0.7.5`。提交同时修改 `.ai/hooks/*` 与 `assets/hooks/*`，说明作者考虑了 source/runtime 双路径，但这不等于 rebase 后或 npm tarball 中仍保持 parity。([GitHub][10])
* **Why it matters:** 默认行为变化和四阶段 hook 接入超出普通 patch 风险边界。直接落入已准备好的 `0.7.5` 会让版本语义、升级预期和回滚边界不清楚。
* **Concrete fix / verification:** 建议独立 `0.8.0` minor release。若坚持 patch，则必须将全部新行为改成显式 opt-in/default-off，并使用新的 `0.7.6`，而不是修改既有 `0.7.5` 内容。对 packed tarball 做 clean-install、adopt、migrate、scaffold、central-first runtime 和 repo-pinned runtime smoke。
