> **Superseded direction (2026-06-21)**: 不再实施多个 MCP URL 或独立 `reader` profile 方案。标准设计是一个 ChatGPT Connector URL `https://.../mcp`，planner 通过 `workspaceReader` capability 暴露 reader tools；`repo-harness adopt` / `repo-harness init` / user-scope setup 将 adopted repo 写入 `~/.repo-harness/registered-repos.json`，MCP 通过 registry 发现 repo，并通过 `repo_path` 选择目标。本文下方保留为研究材料；执行 authority 是 `plans/sprints/20260621-repo-harness-mcp-reader-hardening-sprint.md`。

## 结论

`repo-harness` 不能直接替代 CodexPro，核心不是“某个读取函数坏了”，而是两者的设计目标不同：

* **CodexPro 是通用的、受限的本机工作区文件桥**：ChatGPT 通过 HTTPS MCP 端点连接本机服务，再调用 `open workspace → tree/search → 分段读取`。
* **repo-harness 的 MCP 是工作流侧车**：默认只暴露计划、任务、handoff、docs 等固定工作流文件，并不允许任意读取项目源码或本机文档。它的 full-disk 模式虽然扩大了读取范围，却同时清空了秘密文件拒绝规则，安全上不能作为正式修复。([GitHub][1])

因此正确修复方向是：**给 repo-harness 增加独立的 `reader` 模式，并重构成 CodexPro 类似的 allowed-roots 工作区模型；同时修 OAuth、会话生命周期和工具发现问题。**

# 一、CodexPro 实际是怎样让 GPT 读取本机文档的

它不是让 ChatGPT 直接访问 `localhost` 或本机磁盘，而是下面这条链路：

```text
ChatGPT
  │
  │ HTTPS / Secure MCP Tunnel
  ▼
公网可访问的 /mcp
  │
  ├─ URL Token / Bearer Token 校验
  ├─ MCP Session 管理
  ▼
本机 CodexPro MCP Server
  │
  ├─ open_workspace
  ├─ tree / search
  ├─ read_text
  ▼
allowedRoots + realpath + symlink 检查
  │
  ▼
受限读取本机文件
```

截至 2026 年 6 月 21 日，ChatGPT 官方仍要求连接远程可访问的 MCP 服务；本机私有服务需要稳定 HTTPS 地址或 Secure MCP Tunnel。Pro 当前主要支持 read/fetch 类 MCP 能力，完整写入和修改能力的计划支持范围不同。([OpenAI Help Center][2])

CodexPro 的关键做法有六个。

### 1. 把鉴权简化为“URL 中携带私有 Token”

CodexPro 启动时生成随机 Token，将其加入 MCP Server URL；在 ChatGPT App 配置中选择 `Authentication: None`。服务端实际上仍会校验 URL query token，也支持 Bearer token，并使用常量时间比较避免简单的时序泄漏。([GitHub][1])

这绕开了很多自定义 OAuth 的兼容性问题。不过 query token 可能出现在代理访问日志和浏览器历史中，所以它更适合作为单用户兼容模式，生产环境仍应优先使用完整 OAuth 或官方安全隧道。

### 2. 不是开放整个磁盘，而是先打开工作区

CodexPro 维护 `allowedRoots`。用户请求打开某个目录时，服务端会：

```text
输入目录
→ path.resolve
→ realpath
→ 检查是否位于 allowedRoots
→ 生成工作区
```

后续工具都基于该工作区和相对路径访问文件，而不是让模型不断传递绝对路径。([GitHub][3])

### 3. 同时检查路径字符串和真实文件路径

它不只检查 `../`，还会检查：

* 解析后的绝对路径；
* 文件自身的 `realpath`；
* 不存在文件的最近现存父目录；
* 符号链接是否跳出工作区；
* 是否匹配 `.env`、私钥、SSH key 等阻止规则。

这避免了：

```text
workspace/link -> /Users/me/.ssh
workspace/link/id_rsa
```

这类 symlink escape。([GitHub][3])

### 4. 提供模型容易使用的文件工具

CodexPro 不是只有一个“读文件”工具，而是提供了符合模型探索顺序的工具：

```text
open_workspace
tree
search
read_text
```

`read_text` 支持起止行、最大字节数、截断标记、文件哈希和行号，而不是一次返回整个大文件。([GitHub][1])

### 5. MCP Session 生命周期更完整

其 HTTP MCP 层包括：

* 仅在 `initialize` 时创建 Session；
* 校验 Session ID；
* GET、POST、DELETE `/mcp`；
* `Mcp-Session-Id` 暴露；
* `onclose` 回收；
* Session TTL；
* 最大 Session 数量；
* 定期清理过期 Session。([GitHub][4])

### 6. 控制默认工具数量

CodexPro 提供 minimal、standard、full 等工具模式。小而稳定的默认工具列表能降低模型选错工具以及 ChatGPT 工具扫描缓存不一致的概率。([GitHub][1])

# 二、repo-harness 当前为什么会“连接了但读不到”

## P0：默认 profile 根本不允许读取普通项目文件

`repo-harness` 的 planner policy 默认读取范围主要是：

```text
AGENTS.md
CLAUDE.md
SKILL.md
docs/**
spec/**
plans/**
tasks/**
.ai/context/**
handoff/**
checks/**
```

它并不是：

```text
src/**
app/**
packages/**
任意本机文档
```

所以即使 OAuth 和 MCP Session 都完全正常，GPT 请求读取 `src/index.ts`、`package.json` 或任意 Documents 文件，策略也可能直接拒绝。([GitHub][5])

这很可能是最主要的功能性原因。

---

## P0：full-disk 模式的安全模型有严重缺陷

当前逻辑近似于：

```ts
readGlobs: fullDiskRead ? ["**"] : plannerReadGlobs,
denyGlobs: fullDiskRead ? [] : commonDenyGlobs,
allowAbsoluteRead: fullDiskRead,
```

也就是说，启用 full-disk 后，不只是允许所有读取，还把下面这些拒绝规则一起移除了：

```text
.env
*.pem
*.key
.ssh/**
credentials/**
secrets/**
.git/**
```

这不是可接受的“本机文档读取方案”。广泛读取模式可以扩大允许根目录，但**绝不能清空秘密文件 deny list**。([GitHub][5])

---

## P0：repo-harness 的工具表面仍然是工作流工具

它主要暴露：

```text
harness_status
harness_doctor
discover_harness_repos
list_workflow_files
read_workflow_file
...
```

`read_workflow_file` 虽然最终会读取文件，但：

* 名称会让模型误认为它只能读工作流文件；
* 发现文件依赖固定扫描目录；
* 没有标准的 workspace/tree/search 探索流程；
* 通常一次性读取整个文件；
* 超过 `maxFileBytes` 就失败；
* 不支持 `start_line`、`end_line` 分段读取。([GitHub][6])

因此，即使策略放行了，GPT 也不容易可靠地发现和读取目标文件。

---

## P0：OAuth 实现和 ChatGPT 当前要求存在偏差

repo-harness 确实实现了 refresh token，但 OAuth discovery 中只公布了：

```json
"scopes_supported": ["repo-harness"]
```

没有公布 `offline_access`，授权码和 token 中的 scope 又基本固定为 `repo-harness`。ChatGPT 官方要求 OAuth 服务在 discovery 中公布 `offline_access` 并提供 refresh token，否则首次授权可能成功，但 access token 过期后连接会丢失。([GitHub][7])

此外，公网 origin 默认可能根据：

```text
x-forwarded-proto
x-forwarded-host
```

推导。经过 Cloudflare、ngrok、反向代理链或多层 tunnel 时，只要 Host、Proto 或代理信任设置不一致，OAuth metadata 中的 issuer、authorization endpoint、token endpoint 就可能与 ChatGPT 实际访问地址不同。([GitHub][7])

---

## P1：Session 管理不够健壮

repo-harness 当前基本使用普通 Map 保存 transport：

```ts
Map<sessionId, transport>
```

但缺少 CodexPro 中的：

* Session ID 格式验证；
* TTL；
* 最大 Session 数；
* 周期清理；
* DELETE `/mcp`；
* transport `onclose` 自动删除；
* 更明确的 stale-session 错误。

因此本机服务重启、隧道断线、ChatGPT 保留旧 Session ID 后，比较容易陷入持续的：

```text
No valid session
```

repo-harness 当前实现主要处理 GET 和 POST，而 CodexPro 实现了完整的 GET、POST、DELETE 生命周期。([GitHub][7])

---

## P1：ChatGPT 可能仍在使用旧工具 Schema

repo-harness 文档本身已经提醒：服务器工具发生变化后，ChatGPT App 中扫描到的工具定义可能不会自动更新；仅仅重新选择 App 不等于重新执行 `tools/list` 扫描。([GitHub][8])

所以常见现象是：

```text
本地代码已经增加工具
→ ChatGPT 仍使用旧参数 Schema
→ 调用不存在的工具或旧字段
→ MCP 返回 invalid arguments / unknown tool
```

---

# 三、推荐的最终架构

不要继续扩展现有 planner profile，而是将服务拆成三个明确模式：

```text
reader
  通用只读文件访问
  可暴露给 ChatGPT Pro
  默认公网 MCP 模式

planner
  reader + plans/tasks/handoff 写入
  是否暴露写工具取决于 ChatGPT 计划支持

orchestrator
  shell / git / agent orchestration
  默认仅本机，不通过公网 MCP 暴露
```

建议每个模式使用独立 endpoint 或独立 App：

```text
https://mcp.example.com/reader/mcp
https://mcp.example.com/planner/mcp
```

不要把几十个工作流、浏览器控制、shell 和文件读取工具都塞进同一个 `tools/list`。

# 四、详细修复方案

## 第一部分：修正访问策略

修改：

```text
src/cli/mcp/policy.ts
```

增加 `reader` profile：

```ts
export const READER_READ_GLOBS = ["**"];

export function createReaderPolicy(
  allowedRoots: string[],
): McpPolicy {
  return {
    profile: "reader",

    allowedRoots,

    readGlobs: READER_READ_GLOBS,
    writeGlobs: [],

    // 任何模式都必须保留
    denyGlobs: [
      ...COMMON_DENY_GLOBS,
      "**/.env",
      "**/.env.*",
      "**/*.pem",
      "**/*.key",
      "**/.ssh/**",
      "**/credentials/**",
      "**/secrets/**",
      "**/.git/**",
      "**/node_modules/**",
    ],

    allowAbsoluteRead: false,
    maxFileBytes: 180_000,
  };
}
```

必须删除这种行为：

```diff
- denyGlobs: fullDiskRead ? [] : commonDenyGlobs
+ denyGlobs: [...commonDenyGlobs]
```

将 `fullDiskRead` 改名为更准确的：

```text
broadRead
```

并使它只影响允许根目录，不影响拒绝规则。

## 第二部分：引入 allowedRoots 和 WorkspaceManager

修改：

```text
src/cli/mcp/paths.ts
```

不要再让每个工具接收任意：

```json
{
  "repo_path": "/Users/me/project",
  "path": "/Users/me/project/src/index.ts"
}
```

改为两阶段：

```json
{
  "root": "/Users/me/Projects/demo"
}
```

返回：

```json
{
  "workspace_id": "ws_7f3...",
  "root": "/Users/me/Projects/demo"
}
```

后续只接收：

```json
{
  "workspace_id": "ws_7f3...",
  "path": "src/index.ts"
}
```

路径解析建议统一为：

```ts
async function resolveWorkspacePath(
  workspaceRoot: string,
  requestedPath: string,
): Promise<string> {
  if (path.isAbsolute(requestedPath)) {
    throw new Error("Absolute paths are not accepted");
  }

  const candidate = path.resolve(workspaceRoot, requestedPath);
  assertLexicallyInside(workspaceRoot, candidate);

  const realRoot = await fs.realpath(workspaceRoot);

  try {
    const realCandidate = await fs.realpath(candidate);
    assertRealpathInside(realRoot, realCandidate);
    return realCandidate;
  } catch (error) {
    if (!isNotFound(error)) throw error;

    const existingParent = await findNearestExistingParent(candidate);
    const realParent = await fs.realpath(existingParent);
    assertRealpathInside(realRoot, realParent);

    return candidate;
  }
}
```

跨平台 containment 不要只使用：

```ts
candidate.startsWith(root + path.sep)
```

应使用 `path.relative`：

```ts
function isInside(root: string, candidate: string): boolean {
  const relative = path.relative(root, candidate);

  return (
    relative === "" ||
    (!relative.startsWith(`..${path.sep}`) &&
      relative !== ".." &&
      !path.isAbsolute(relative))
  );
}
```

Windows 还需要测试：

* 大小写不同的盘符；
* `C:\` 与 `D:\`；
* UNC 路径；
* junction；
* symlink；
* `\\?\` 路径。

## 第四部分：修 OAuth

修改：

```text
src/cli/mcp/oauth.ts
src/cli/mcp/transports/http.ts
```

### 1. 公布 offline_access

```diff
- scopes_supported: ["repo-harness"]
+ scopes_supported: ["repo-harness", "offline_access"]
```

授权时保存真实请求的 scope：

```ts
const requestedScopes = parseScopes(req.query.scope);

const allowedScopes = requestedScopes.filter((scope) =>
  ["repo-harness", "offline_access"].includes(scope)
);

authCodes.set(code, {
  clientId,
  redirectUri,
  codeChallenge,
  scopes: allowedScopes,
});
```

Token 返回：

```ts
{
  access_token: accessToken,
  token_type: "Bearer",
  expires_in: ACCESS_TOKEN_TTL_SECONDS,
  refresh_token: refreshToken,
  scope: storedScopes.join(" "),
}
```

### 2. Refresh 时轮换 access token

当前不应继续复用原 access token。改为：

```text
验证旧 refresh token
→ 吊销旧 access token
→ 吊销旧 refresh token
→ 生成新 access token
→ 生成新 refresh token
→ 原子写入存储
```

### 3. 强制配置公网 Origin

生产模式不要依赖转发头推导：

```bash
REPO_HARNESS_MCP_PUBLIC_ORIGIN=https://mcp.example.com
```

启动时验证：

```ts
if (!isLoopback(host) && !process.env.REPO_HARNESS_MCP_PUBLIC_ORIGIN) {
  throw new Error(
    "REPO_HARNESS_MCP_PUBLIC_ORIGIN is required for public MCP endpoints",
  );
}
```

环境变量必须只包含 origin：

```text
正确：https://mcp.example.com
错误：https://mcp.example.com/mcp
```

### 4. 不要硬编码 ChatGPT 回调地址

应以 Dynamic Client Registration 中保存的 `redirect_uris` 为准：

```text
authorize 请求 redirect_uri
必须精确匹配该 client 注册过的 URI
```

可以再叠加 HTTPS 和域名 allowlist，但不要仅靠写死某两个 ChatGPT 域名和路径，否则平台地址调整后会突然失效。

### 5. 给 OAuth endpoint 加限流

至少对这些路径限流：

```text
/authorize
/register
/token
/revoke
```

尤其公开 tunnel 模式下，不能关闭 token endpoint rate limit。

## 第五部分：增加 CodexPro 类兼容鉴权模式

为单用户场景增加：

```text
--auth url-token
```

启动时生成 32 字节随机 token：

```ts
const token = crypto.randomBytes(32).toString("base64url");
```

接受两种形式：

```http
Authorization: Bearer <token>
```

或：

```text
https://mcp.example.com/mcp?repo_harness_token=<token>
```

比较使用：

```ts
crypto.timingSafeEqual()
```

安全约束：

* 非 loopback 地址无 token 时拒绝启动；
* 访问日志必须移除 query string；
* `/health` 不得返回 token；
* CLI 输出只完整显示一次；
* 后续日志只显示脱敏值；
* 提供 token rotate；
* OAuth 仍作为推荐生产模式。

这一模式可以用于判断问题究竟在 OAuth，还是文件 policy/tool 层：

```text
URL Token 能读，OAuth 不能读
→ OAuth/metadata 问题

URL Token 也不能读
→ policy/path/tool schema 问题
```

## 第六部分：重构 MCP Session 管理

修改：

```text
src/cli/mcp/transports/http.ts
```

使用：

```ts
interface SessionRecord {
  transport: StreamableHTTPServerTransport;
  createdAt: number;
  lastSeenAt: number;
}

const sessions = new Map<string, SessionRecord>();

const SESSION_TTL_MS = 30 * 60 * 1000;
const MAX_SESSIONS = 64;
```

每次请求：

```text
校验 UUID
→ 清理过期 session
→ 更新 lastSeenAt
→ 交给已有 transport
```

只允许在没有 Session ID 且请求是 `initialize` 时创建 Session。

增加：

```ts
app.post("/mcp", auth, handleMcpRequest);
app.get("/mcp", auth, handleMcpRequest);
app.delete("/mcp", auth, handleMcpRequest);
```

并在 transport 关闭时：

```ts
transport.onclose = () => {
  sessions.delete(sessionId);
};
```

未知 Session 的响应应机器可读：

```json
{
  "error": {
    "code": "SESSION_NOT_FOUND",
    "message": "The MCP session is missing or expired; initialize a new session."
  }
}
```

不要把带旧 Session ID 的任意请求自动当作新 initialize，否则容易产生协议状态混乱。

## 第七部分：补齐 HTTP 兼容层

建议增加：

```ts
app.use(cors({
  origin: configuredAllowedOrigins,
  methods: ["GET", "POST", "DELETE", "OPTIONS"],
  allowedHeaders: [
    "Content-Type",
    "Accept",
    "Authorization",
    "Mcp-Session-Id",
    "MCP-Protocol-Version",

  exposedHeaders: ["Mcp-Session-Id"],

```

服务器到服务器的 ChatGPT 调用不一定依赖浏览器 CORS，但增加明确的 OPTIONS、header 和 DELETE 支持可以改善 Inspector、Web 管理界面及其他 MCP 客户端的互操作性。不要把 `Access-Control-Allow-Origin: *` 与带凭据 OAuth 混用。

不要返回 allowed root 的完整敏感路径；可以只返回数量或脱敏名称。

# 六、立即可以执行的排查步骤

在修改代码前，先不要使用：

```text
--repo /
--allow-full-disk-read
```

改为单仓库运行，并显式设置稳定公网 origin：

```bash
export REPO_HARNESS_MCP_PUBLIC_ORIGIN="https://mcp.example.com"

repo-harness mcp serve \
  --repo "/absolute/path/to/project" \
  --transport http \
  --host 127.0.0.1 \
  --port 8765 \
  --profile planner
```

先检查本机健康状态：

```bash
curl -fsS http://127.0.0.1:8765/health
```

检查公网 OAuth discovery：

```bash
curl -fsS \
  https://mcp.example.com/.well-known/oauth-authorization-server
```

重点检查：

```json
{
  "issuer": "https://mcp.example.com",
  "authorization_endpoint": "https://mcp.example.com/...",
  "token_endpoint": "https://mcp.example.com/...",
  "scopes_supported": [
    "repo-harness",
    "offline_access"
  ]
}
```

再检查 protected resource metadata：

```bash
curl -fsS \
  https://mcp.example.com/.well-known/oauth-protected-resource/mcp
```

其中的 `resource` 必须与 ChatGPT 实际连接的公网 MCP origin 和路径一致。

代码或工具 Schema 修改后必须：

```text
重启本机服务
→ 确认 tunnel 仍指向新进程
→ 在 ChatGPT App 中重新扫描/刷新工具
→ 必要时删除并重建 App
→ 新建对话
→ 明确选择该 App
```

仅在原对话中重新选择工具，并不能保证 ChatGPT 丢弃旧 Schema。([GitHub][8])

## 修复优先级

首先修改 `policy.ts`：新增 `reader`，彻底移除 full-disk 清空 deny list 的行为。

其次修改 `tools.ts` 和 `paths.ts`：实现 `allowedRoots + workspace_id + tree/search/read_text`。

然后修 `offline_access`、固定公网 origin 和 refresh token rotation。

最后补 Session TTL、DELETE、CORS、schema hash 和完整测试。

按这个顺序处理后，`repo-harness` 会从“只适合计划文件的工作流侧车”，变成与 CodexPro 类似、但安全边界更清晰的通用本机只读 MCP 服务。由于目前没有你的实际服务日志和 ChatGPT 返回错误，不能断定某一项是唯一触发点；但从代码结构看，**默认 policy 范围不匹配、OAuth metadata 和工具 Schema 缓存**是最应优先验证的三项。
